# V0.2.0 Agent Runtime / Trace Research

本报告整理第一轮开源 Agent 项目研究结果，目标是为
Agent Harness Template 后续 Runtime、Trace、Step、Event 和 Checkpoint
设计提供参考。

本阶段只做研究，不引入新功能，不创建业务 Agent。

## Runtime 设计摘要

### LangGraph

LangGraph 的核心是 Graph Runtime：`State + Node + Edge`。

- State 是共享运行状态。
- Node 是执行函数。
- Edge 决定下一步执行路径。
- 执行模型支持多个节点逐步推进，适合长时运行和可恢复 workflow。

它的 persistence 分为：

- checkpointer：保存 thread 级 graph state，适合恢复、中断和状态回看。
- store：保存跨 thread 的长期数据。

适合借鉴：

- Run 内部引入 state snapshot。
- 每个 Step / Node 完成后可保存 checkpoint。
- Event 和 Step 明确挂到 Node 执行过程。

暂时不适合：

- 完整 super-step 调度。
- time travel。
- human interrupt。
- 复杂 subgraph。

### OpenAI Agents SDK

OpenAI Agents SDK 的 Runner 是循环式 Agent Runtime。

典型流程：

1. 从 starting agent 开始。
2. 模型输出 final output 时结束。
3. 如果发生 handoff，则切换 agent。
4. 如果发生 tool call，则执行 tool，再进入下一轮。

它的 tracing 设计值得重点参考：

- 一次 run 对应一个 trace。
- Agent run、LLM generation、function tool call、guardrail、handoff 都可以是 span。
- span 包含 trace id、span id、parent span id、开始时间、结束时间和 span data。

它的 RunItem 设计也很有启发：

- message
- tool call
- tool output
- handoff
- reasoning

这些 item 是结构化序列，而不是纯文本日志。

适合借鉴：

- 增加 `trace_id`、`span_id`、`parent_span_id`。
- Event 从纯日志升级为 typed event。
- Tool Call 和 Tool Result 单独建模。
- Run Result 明确包含 final output、items、usage、errors。

暂时不适合：

- handoff。
- guardrails。
- hosted tools。
- human-in-the-loop。
- sandbox agent。

### Microsoft AutoGen

AutoGen 的 AgentChat 使用 message 驱动。

它区分：

- agent-agent messages。
- agent 内部 events。

内部事件包括：

- ToolCallRequestEvent。
- ToolCallExecutionEvent。
- ToolCallSummaryMessage。

Agent 支持：

- `run`：返回 TaskResult。
- `run_stream`：流式产出 BaseAgentEvent / BaseChatMessage，最后产出 TaskResult。

适合借鉴：

- 把 Event 分为外部可见 message 和内部 execution event。
- Tool call request / execution / summary 拆成不同 event type。
- 后续可设计 streaming event API。

暂时不适合：

- 多 Agent group chat。
- swarm / selector group chat。
- 复杂 agent-to-agent 协议。
- multimodal message。

### CrewAI

CrewAI 有两层模型：

- Crews：偏自治协作。
- Flows：偏可控事件驱动 workflow。

Task / Agent / Crew 支持 YAML 配置，这和当前
`module.yaml` / `agent.yaml` / scaffold 方向一致。

Flow 的状态管理值得参考：

- 支持非结构化 state。
- 支持 Pydantic structured state。
- 每个 state 自动有唯一 ID。
- 支持持久化 Flow state。

Tool 规范也较清晰：

- name。
- description。
- args schema。
- run 方法。

适合借鉴：

- YAML-first 配置风格。
- structured state。
- flow state id。
- tool args schema 作为标准契约。

暂时不适合：

- Crew 协作模型。
- 大量工具生态。
- 可视化 flow plot。
- 外部 observability vendor 集成。

## 和当前 agent-harness-template 的对比

### 已有能力

- Run / Step / Event 最小模型。
- PostgreSQL 持久化。
- demo_agent LangGraph 最小状态机。
- Module Registry + Agent Execution Contract。
- Skill / Tool Registry。
- Provider Router。
- File / Artifact。
- 简单文本知识检索和 citation。
- scaffold module。
- 测试与业务无关防污染机制。

### 当前缺口

- Event 仍偏日志，没有形成统一 typed event contract。
- Step 缺少 `parent_step_id`、`trace_id`、`span_id`、`attempt`、`started_at`、`ended_at`、`error`。
- Tool Call 没有独立模型，目前 tool 结果只是 step output 的一部分。
- Run 缺少 `trace_id`、`module_id`、`agent_id`、`workflow_id`、`usage`、`error`。
- Checkpoint 还没有独立模型，不能恢复到某个节点之后。
- Eval 还只是目录和文档，没有针对 trace / trajectory 的评价。
- Observability 只有事件查询，没有 trace tree / timeline / span 视图。

### 不适合现在引入的复杂设计

- LangGraph 完整 checkpoint / time travel。
- AutoGen 多 Agent group chat。
- OpenAI Agents SDK 的 handoff、guardrails、human-in-the-loop。
- CrewAI 的完整 Crew 协作系统。
- 外部 observability 平台集成。
- 生产级 retry policy / distributed worker。
- 多租户权限模型。

## 推荐吸收的设计

### 适合 V0.2.1

- 引入 Trace / Span 概念，但只做数据模型和 API 返回，不接外部观测平台。
- 扩展 Run / Step / Event 字段：
  - `trace_id`
  - `span_id`
  - `parent_span_id`
  - `sequence`
  - `event_type`
  - `status`
  - `started_at`
  - `ended_at`
  - `duration_ms`
  - `error`
  - `metadata`
- 将 Step 规范化为 node / tool / llm / final 等类型。
- 增加 Tool Call Event 类型，但不做复杂工具执行权限。

### 适合后续版本

- V0.2.2：Checkpoint 最小版，每个 node 完成后保存 state snapshot。
- V0.2.3：失败、重试、恢复最小版，支持失败 step 标记、retry count、resume from checkpoint。
- V0.2.4：Trace timeline API 和前端 timeline 视图。
- V0.2.5：Eval trajectory，基于 run events、tool calls、final output 做离线评价。

### 暂时不要做

- 完整多 Agent handoff。
- HITL。
- 权限系统。
- 外部 tracing vendor。
- 复杂 workflow planner。
- 自动 tool repair。
- 多模态消息模型。

## 小步改进路线

### V0.2.1：Trace / Event Contract 增强

目标：让每次 Run 可解释、可画 timeline、可 debug。

状态：已进入实现阶段，范围限定为 Trace / Span / Typed Event / Step 字段增强。

建议做：

- 新增或完善 schema：
  - `trace.schema.json`
  - `span.schema.json`
  - `tool-call.schema.json`
- 扩展数据库：
  - `run_events` 增加 `trace_id`、`span_id`、`parent_span_id`、`sequence`、`metadata`
  - `steps` 增加 `type`、`started_at`、`ended_at`、`duration_ms`、`error`
- API 保持兼容：
  - `GET /api/runs/{run_id}/events` 继续可用
  - 可新增 `GET /api/runs/{run_id}/trace`
- demo_agent 每个 node 生成 span-like event。
- 不实现 checkpoint。

### V0.2.2：Checkpoint 最小版

目标：让运行状态可以被保存和检查，但先不做复杂恢复。

建议做：

- 新增 `checkpoints` 表：
  - `id`
  - `run_id`
  - `step_id`
  - `checkpoint_index`
  - `state`
  - `created_at`
- 每个 node completed 后保存 state snapshot。
- 新增：
  - `GET /api/runs/{run_id}/checkpoints`
  - `GET /api/checkpoints/{checkpoint_id}`
- 不做 time travel。
- 不做自动 resume。

### V0.2.3：失败 / 重试 / 恢复最小版

目标：Run 不再只有 happy path，失败可定位，简单重试可追踪。

建议做：

- Step 增加：
  - `attempt`
  - `max_attempts`
  - `error_type`
  - `error_message`
- Event 增加：
  - `step.failed`
  - `step.retry_scheduled`
  - `step.retried`
  - `run.failed`
- 新增最小 retry policy：
  - 只支持固定次数。
  - 默认关闭。
- 可新增：
  - `POST /api/runs/{run_id}/retry`
- 从最后 checkpoint 恢复可放到更后版本，不建议一次完成。

## 结论

当前底座已经有 Run / Step / Event、PostgreSQL、Module Registry、
Provider、Skill / Tool Registry 等关键基础。下一步最值得做的不是让
Agent 更复杂，而是让 Runtime 更可观察、更可恢复。

建议 V0.2.1 优先做 Trace / Span / Typed Event Contract。这一步收益高、
风险低，也不会推翻当前架构。
